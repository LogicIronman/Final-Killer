# 期末杀手 Phase 3 开发日志

## 阶段结论

本阶段完成正式复习系统，补上项目最重要的学习闭环：

```text
新题答错
  -> 进入复习队列
  -> 按优先级重复复习
  -> 连续答对 3 次
  -> 进入已完成
```

同时增加已完成题目列表、题目详情、错误历史和单题重置。登录用户使用 SQLite 持久化，游客使用 localStorage，并保持相同的状态转换规则。

## 阶段编号说明

原始提示词将“复习系统”定义为 Phase 2，将筛选、图表、导入导出等体验增强定义为 Phase 3。

实际开发过程中，第二阶段先完成了上一题回看、快捷键、重点题和排行榜，导致阶段编号与原始提示词错位。Phase 2 开发日志已经把“正式复习模式”列为下一阶段最高优先级。

因此本阶段没有继续叠加图表或深色模式，而是优先补齐核心复习闭环。这样可以避免产品已经具备排行榜和重点题，却仍不能真正复习错题的问题。

## 本阶段目标

1. 从 `reviewing` 状态中抽取复习题。
2. 连续答对次数更低的题优先出现。
3. 同连续次数下，最后答错时间更早的题优先。
4. 复习答错后连续答对次数归零。
5. 连续答对 3 次后进入 `done`。
6. 登录用户和游客使用一致规则。
7. 增加已完成列表。
8. 增加题目详情、正确答案、解析和错误历史。
9. 支持重置单题进度。

## 核心业务规则

### 状态转换

```text
unseen
  -> 新题答对 -> done
  -> 新题答错 -> reviewing

reviewing
  -> 复习答对且连续次数 < 3 -> reviewing
  -> 复习连续答对 3 次 -> done
  -> 复习答错 -> reviewing，连续次数归零

done
  -> 单题重置 -> unseen
```

### 复习队列优先级

服务器排序规则：

1. `consecutive_correct` 升序。
2. `last_wrong_at` 升序。
3. 同优先级内随机。

这意味着掌握程度最低的题优先；掌握程度相同时，等待复习更久的题优先。

游客端采用相同逻辑，并为旧 localStorage 数据提供 `lastAnsweredAt` 回退，避免已有游客进度失效。

### 单题重置语义

重置会删除该用户在该题上的 `user_progress` 记录，因此同时清除：

- 当前状态。
- 累计答题次数。
- 累计正确和错误次数。
- 连续答对次数。
- 错误历史。
- 重点标记。

重置不会回滚 `daily_logs`。每日统计表示真实发生过的历史活动，不应因用户重新学习一道题而被改写。

## 数据库变化

### 新增字段

`user_progress` 增加：

```sql
last_wrong_at TEXT
```

用途：支持按最后答错时间稳定排序复习队列。

### 兼容迁移

数据库初始化仍使用幂等 `CREATE TABLE IF NOT EXISTS`。对于已经存在的 Phase 1/2 数据库，启动时读取：

```sql
PRAGMA table_info(user_progress)
```

如果缺少 `last_wrong_at`，自动执行：

```sql
ALTER TABLE user_progress ADD COLUMN last_wrong_at TEXT;
```

该迁移不会删除用户、题库或学习进度，不需要重新执行会清空进度的 seed。

## 后端实现

### 新增接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/questions/review?limit=10` | 按优先级获取复习题 |
| GET | `/api/questions/:id` | 获取题目详情、正确答案、进度和错误历史 |
| GET | `/api/progress/completed` | 获取已完成题目 |
| POST | `/api/progress/reset` | 重置单题进度 |

以上接口均需要 JWT。

### 已有接口增强

`POST /api/progress/answer` 原有 `mode: "review"` 逻辑正式接入页面，并在答错时同步更新：

- `last_wrong_at`
- `wrong_history_json`
- `wrong_count`
- `consecutive_correct`

复习题响应额外包含：

```json
{
  "reviewProgress": {
    "consecutiveCorrect": 1,
    "wrongCount": 2
  }
}
```

前端可以在提交前展示当前连续答对进度。

## 游客模式实现

游客数据结构新增：

- `lastWrongAt`
- `wrongHistory`

同时将游客状态扩展为：

```text
unseen | reviewing | done
```

旧数据兼容策略：

- 缺少 `wrongHistory` 时按空数组处理。
- 缺少 `lastWrongAt` 时使用 `lastAnsweredAt` 排序。
- 仅重点标记、尚未答题的记录使用 `unseen`。
- 不要求用户清理 localStorage 或重新开始。

新增游客函数：

- 获取复习队列。
- 获取已完成题目。
- 获取题目详情。
- 重置单题进度。

## 前端实现

### 复习模式

路由：

```text
/practice/review
```

复习页复用新题模式组件，没有复制第二套刷题页面。共享能力包括：

- 单选、多选、判断题。
- 上一题和答题状态回看。
- `A-E` 与 `1-5` 选择。
- Enter 提交或进入下一题。
- 左右方向键导航。
- 重点标记。
- 正确、错误反馈和解析。

模式差异集中在：

- 抽题接口。
- 答题提交的 `mode`。
- 连续答对进度。
- 状态规则文案。
- 队列为空时的下一步操作。

连续答对区域使用 3 段固定进度条，同时显示数字 `x / 3`，不会仅依赖颜色表达状态。

### 已完成列表

路由：

```text
/completed
```

功能：

- 展示所有 `done` 题目。
- 显示章节、题型和历史错误次数。
- 进入题目详情。
- 空状态可以返回新题模式。

### 题目详情

路由：

```text
/question/:id
```

展示：

- 题干和全部选项。
- 正确答案高亮及文字标识。
- 题目解析。
- 当前学习状态。
- 累计答题、答对、答错和连续答对次数。
- 历次错误答案、正确答案和时间。
- 重点状态。

重置采用页面内二次确认，不使用弹窗。确认后立即更新页面为未做状态，并说明每日统计不会回滚。

### 导航与仪表盘

- 顶部导航新增“复习模式”。
- 登录用户阶段标识更新为 Phase 3。
- 仪表盘“未做题”进入新题模式。
- 仪表盘“复习中”进入复习模式。
- 仪表盘“已完成”进入已完成列表。
- 存在复习题时显示“开始复习”主操作。
- 错题本增加“开始复习”和“查看详情与历史”。
- 重点题增加“查看题目详情”。

## 已知限制

- 当前复习会在队列很小时连续出现同一道题，尚未加入最小间隔或间隔重复算法。
- 错误历史只记录错误提交，不记录每一次正确提交。
- 单题重置不会回滚每日历史统计。
- 游客进度不会迁移到登录账号。
- 前端仍缺少组件测试和端到端浏览器测试。
- 已完成列表暂无章节和题型筛选。
- 当前仅有一个正式题库，没有题库切换界面。
